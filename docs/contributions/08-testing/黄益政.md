## 测试策略

Android 端采用分层测试策略，覆盖网络层、仓储层、ViewModel 状态层和 Compose UI 层。

| 测试类型 | 方式 | 说明 |
|---------|------|------|
| 单元测试 | JUnit + kotlinx-coroutines-test + Robolectric | 验证 ViewModel、Repository、ApiManager、ConfigManager、工具类等逻辑 |
| Mock API 测试 | MockWebServer | 模拟后端响应，验证 Retrofit 接口和网络异常处理 |
| Flow / 状态测试 | Turbine + 协程测试工具 | 验证 StateFlow 状态更新、加载态和错误态 |
| Compose UI 测试 | androidx.compose.ui.test | 验证页面渲染、按钮点击、表单输入和核心交互 |
| 覆盖率统计 | JaCoCo + Codecov | 生成 Android 单元测试覆盖率并上传报告 |
| 安全测试 | Gitleaks / Trivy / Semgrep / MobSF / mobsfscan / CodeQL | 检测密钥泄露、依赖漏洞、Android 安全风险和 Kotlin 代码风险 |

## AI 辅助测试

### 使用场景

- 生成 ViewModel 正常流、异常流和空数据场景测试思路
- 补充 ApiManager、MainRepository、ConfigManager 的边界测试
- 梳理 Compose UI 页面中可验证的文本、按钮和点击路径
- 检查协程测试中是否需要 `advanceUntilIdle()`、`Dispatchers.setMain()` 和清理逻辑

### 人工调整

- 根据实际 ViewModel 构造参数和 Repository 接口调整测试代码
- 对需要 Android Context 的测试使用 Robolectric
- 对网络层测试使用 MockWebServer 隔离真实后端
- 对 UI 测试保留稳定文本和明确交互，减少对视觉细节的脆弱依赖

## 遇到的问题和解决

1. **协程测试状态未及时更新**  
   原因：ViewModel 使用 `viewModelScope.launch`，断言时协程可能尚未执行完成。  
   解决：使用 `StandardTestDispatcher`、`runTest` 和 `advanceUntilIdle()` 控制调度。

2. **SharedPreferences 测试依赖 Android Context**  
   原因：`ConfigManager` 使用 Android 本地存储。  
   解决：使用 Robolectric 提供测试 Context，保证 JVM 单元测试可运行。

3. **网络接口测试不能依赖真实后端**  
   原因：真实环境不稳定会导致测试偶发失败。  
   解决：使用 MockWebServer 构造响应，验证 Retrofit 调用、响应解析和错误路径。

4. **Compose UI 测试容易受文案变化影响**  
   原因：UI 测试常通过文本查找节点，文案变化会导致测试失败。  
   解决：优先选择稳定标题、按钮和核心状态文案，避免测试过多装饰性文本。

5. **覆盖率与 UI 代码的关系需要取舍**  
   原因：Compose UI 文件直接纳入覆盖率会导致覆盖率不稳定。  
   解决：覆盖率统计重点放在 ViewModel、Repository、ApiManager 等逻辑层，UI 通过 androidTest 验证核心交互。

## 心得体会

Android 端测试工作的重点不是单纯追求用例数量，而是把容易出错的状态流转、网络错误、会话保存和业务分支稳定下来。通过单元测试验证逻辑层，通过 Compose UI 测试验证关键页面入口，再用 JaCoCo 和 Codecov 建立持续反馈，可以让 Android 端在功能快速增加时仍保持可维护性。AI 咨询、Agent 助手和客服功能的测试也让我意识到，大模型能力接入后，客户端更需要测试异常兜底和状态恢复，因为这些才是用户真正感受到的稳定性。
