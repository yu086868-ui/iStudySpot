## 我完成的工作

### 一、Android 应用架构与基础设施

#### 1.1 应用入口与导航结构

- 搭建 Android 用户端单 Activity + Jetpack Compose 的应用结构
- 使用 Navigation Compose 管理页面跳转，覆盖登录、注册、首页、自习室、座位图、预约、订单、个人中心、更多功能、AI 咨询、Agent 助手、在线客服等页面
- 通过 `NavRoutes.kt` 统一维护路由定义，减少页面跳转中硬编码路径
- 在 `AppNavigation.kt` 中集中装配页面、ViewModel 和跨页面参数传递

#### 1.2 Compose UI 与 Material 3 主题

- 使用 Jetpack Compose 实现主要页面，采用声明式 UI 模式，让界面由状态驱动
- 使用 Material 3 组件体系构建按钮、输入框、卡片、对话框、Snackbar 等交互元素
- 扩展主题色系统，补充 success、warning、info、gradient 等业务语义色
- 支持亮色 / 暗色模式，保证首页、个人中心、AI 角色选择等渐变区域在不同主题下仍具备可读性

#### 1.3 MVVM 状态管理

- 按 MVVM 思路划分 UI、ViewModel、Repository、Network 层
- ViewModel 使用 `StateFlow` 暴露页面状态，Compose 页面订阅状态并渲染
- 使用 Kotlin Coroutines 处理异步请求，避免网络请求阻塞 UI
- 将页面 loading、error、empty、success 等状态集中到 ViewModel 层编排

#### 1.4 网络请求与会话管理

- 使用 Retrofit 定义后端 REST API，使用 OkHttp 作为底层 HTTP 客户端
- 封装 `ApiService`、`ApiManager`、`ApiClient` 三层网络结构：
  - `ApiService` 负责声明接口
  - `ApiManager` 负责请求执行、Mock 分支和错误归一化
  - `ApiClient` 负责 Retrofit / OkHttp 实例、请求头、访问令牌注入和 token refresh 路径
- 通过 `ConfigManager` 使用 SharedPreferences 保存访问令牌、用户标识、昵称和主题偏好
- 通过 `MainRepository` 统一暴露业务用例，降低 ViewModel 对网络细节的耦合

## 遇到的问题和解决

1. **Compose 状态流转较复杂**：多个页面同时涉及 loading、error、empty 和 success 状态。解决方式是将状态集中在 ViewModel 中维护，通过 StateFlow 暴露给 UI，避免页面直接处理网络细节。
2. **座位布局数据不稳定**：座位布局接口可能返回空数据或结构不完整。解决方式是在 `StudyRoomViewModel` 中加入座位布局失败后的普通座位列表回退，保证页面仍能展示可用信息。
3. **AI / Agent 功能协议多样**：AI 咨询、Agent 工具调用、客服对话的请求和响应结构不同。解决方式是分别建立模型和 ViewModel，再通过 Repository 统一接入网络层。
4. **测试与真实接口并行推进**：后端接口尚未完全稳定时，Android 端仍需开发和验证。解决方式是在 `ApiManager` 中保留 Mock 分支，并用 `BuildConfig.USE_MOCK` 控制切换。

## 心得体会

本次 Android 前端开发让我更系统地掌握了 Compose + MVVM 的移动端工程实践。相比传统 XML 页面，Compose 更适合状态驱动的业务界面，但也要求我们更认真地设计状态边界、数据流和错误处理。通过 Repository、ApiManager 和 ViewModel 的分层，Android 端在业务页面不断增加的情况下仍保持了相对清晰的结构。AI 咨询、Agent 助手和客服功能的接入，也让我理解了移动端在大模型能力前面需要承担的体验编排、异常兜底和结果呈现职责。
