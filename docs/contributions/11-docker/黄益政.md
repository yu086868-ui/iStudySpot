## Dockerfile 详细说明

### 多阶段构建设计

我编写的 Android 前端 Dockerfile 采用了三阶段构建：

#### 阶段1: 构建阶段 (builder)
- 基础镜像：`eclipse-temurin:17-jdk-jammy`
- 安装 Gradle 8.5 和 Android SDK
- 配置 Android SDK 组件（platform-tools, platforms;android-34, build-tools;34.0.0）
- 利用 Docker 缓存机制，先复制依赖配置文件并下载依赖
- 最后复制源代码并构建 Debug APK

#### 阶段2: 测试阶段 (tester)
- 基于构建阶段
- 运行单元测试并生成 JaCoCo 覆盖率报告

#### 阶段3: 输出阶段 (output)
- 基础镜像：`eclipse-temurin:17-jre-jammy`（更轻量）
- 仅复制构建产物：
  - `app-debug.apk` - 安装包
  - `jacocoTestReport.xml` - 覆盖率报告
  - 测试报告目录

### .dockerignore 配置

排除了以下类型的文件：
- Gradle 构建输出（`.gradle/`, `build/`）
- IDE 配置文件（`.idea/`, `*.iml`）
- 本地配置文件（`local.properties`）
- 敏感信息文件（`*.jks`, `*.keystore`, `secrets.properties`）
- Git 相关文件（`.git/`）
- CI/CD 配置（`.github/`）
- 文档文件（`*.md`, `docs/`）
- 临时文件和原生构建输出

## 遇到的问题和解决

1. 问题：Android SDK 下载和配置在 Docker 中较为复杂
   解决：使用 Android command-line tools 进行 SDK 安装，并通过 `sdkmanager` 管理组件

2. 问题：Gradle 构建需要大量内存，可能导致 OOM
   解决：配置 `GRADLE_OPTS="-Xmx4g -XX:+HeapDumpOnOutOfMemoryError"` 环境变量

3. 问题：Docker 镜像体积过大
   解决：采用多阶段构建，最终输出镜像仅包含 JRE 和构建产物，大幅减小体积

4. 问题：依赖下载耗时较长
   解决：利用 Docker 缓存机制，先复制依赖配置文件，单独执行 `gradle dependencies`，这样依赖层可以被缓存

## 心得体会

通过本次 Docker 部署工作，我深刻理解了容器化技术在移动应用开发中的应用价值：

1. **环境一致性**：Docker 确保了开发、测试、生产环境的一致性，避免了"在我机器上能跑"的问题。

2. **多阶段构建的优势**：通过多阶段构建，可以在构建阶段使用完整的 JDK 和 Android SDK，而最终镜像只需要 JRE，大大减小了镜像体积。

3. **CI/CD 集成**：Docker 与 GitHub Actions 的结合，实现了自动化的构建、测试和部署流程，提高了开发效率。

4. **安全考量**：通过 .dockerignore 排除敏感文件（如密钥库、本地配置），避免了敏感信息泄露的风险。

5. **缓存优化**：合理利用 Docker 层缓存，将依赖下载和代码构建分离，可以显著加快构建速度。

这次实践让我掌握了 Android 项目 Docker 化的完整流程，对于后续的持续集成和持续部署工作有很大的帮助。
