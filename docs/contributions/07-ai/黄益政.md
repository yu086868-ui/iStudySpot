## 我完成的工作

### 一、Android 端 AI 咨询功能接入

#### 1.1 AI 角色与聊天模型

- 定义 Android 端 AI 咨询所需数据模型，覆盖角色、消息、请求和响应结构
- 支持用户消息与 AI 消息区分，便于聊天列表按不同样式展示
- 在 Android 端内置默认角色：
  - 学霸猫：偏学习规划和方法论
  - 温柔学姐：偏耐心解释和情绪陪伴
  - 严厉导师：偏目标管理和效率提醒
- 支持 `sessionId` 会话保持，为多轮对话预留上下文衔接能力

#### 1.2 AI 咨询 ViewModel

- 实现 `AiChatViewModel`，负责 AI 聊天状态编排
- 使用 `StateFlow` 暴露消息列表、加载状态、角色列表和当前选中角色
- 用户发送消息后先追加用户消息，再异步请求 AI 回复
- 请求失败时返回友好兜底文案，避免页面空白或崩溃
- 支持清空消息并重置会话状态

#### 1.3 AI 咨询页面

- 实现 AI 聊天页面，展示用户消息、AI 回复、输入框和发送按钮
- 实现角色选择入口，让用户在不同 AI 角色之间切换
- 在请求中展示加载状态，提示用户 AI 正在回复
- 将 AI 咨询入口接入首页和导航系统，使用户可以从主流程进入智能咨询

#### 1.4 Mock 与真实接口切换

- 通过 `BuildConfig.USE_MOCK` 支持 Mock 回复与真实后端接口切换
- Mock 回复覆盖预约、签到、开放时间、取消规则、价格、使用规则等高频问题
- 真实接口通过 `MainRepository -> ApiManager -> ApiService` 调用 `/api/chat`
- 保留 `characterId` 和 `sessionId` 参数，便于与后端角色对话系统对齐

### 二、Android 端 Agent 助手功能接入

#### 2.1 Agent 数据模型与工具结果展示

- 定义 Agent 对话、工具目录、工具执行结果、回复 block、UI action 等数据结构
- 支持后端返回多个工具结果，并在前端进行折叠和摘要展示
- 对工具名称进行中文展示映射，如自习室、座位、我的预约、预约规则等

#### 2.2 AgentViewModel 状态编排

- 实现 `AgentViewModel`，负责工具目录加载、会话状态、消息列表和建议问题
- 支持加载 `/api/agent/tools/catalog` 工具目录
- 支持调用 `/api/agent/chat` 发送自然语言问题
- 支持调用 `/api/agent/tools/execute` 直接执行工具
- 对 401、空消息、缺少自习室编号、不支持工具等错误进行可读化处理

#### 2.3 多会话与本地会话管理

- 实现 Agent 多会话切换能力
- 支持新建、打开、关闭、删除、置顶会话
- 通过 `AgentConversationStore` 保存本地会话快照、消息摘要和建议问题
- 限制本地保存消息数量，避免历史会话无限增长

#### 2.4 工具快捷入口与建议问题

- 为常用工具提供快捷入口：
  - 查看可用自习室
  - 查看预约规则
  - 查看我的预约
  - 查看指定自习室座位
- 根据工具返回结果动态生成下一步建议问题
- 根据工具结果中的自习室 ID 推断后续座位查询参数，减少用户重复输入

### 三、Android 端在线客服功能接入

#### 3.1 客服欢迎语与推荐问题

- 实现 `CustomerServiceViewModel`
- 调用后端欢迎语接口，加载欢迎文案和推荐问题
- 推荐问题可作为用户快速提问入口，降低首次使用成本

#### 3.2 客服聊天流程

- 支持用户发送客服消息并追加到消息列表
- 使用独立 `sessionId` 维持客服会话
- 调用 `customerServiceChat` 获取客服回复
- 网络异常时展示“网络异常，请稍后重试”等兜底提示

### 四、学习卡片 AI 结果展示入口

- Android 端接入卡片列表接口 `/api/card/list`
- 定义卡片列表展示模型，包含 uuid、稀有度、边框主题、卡面主题、主题分类、Markdown 内容、学习时长和创建时间
- 通过 `CardViewModel` 和卡片收藏页面展示用户学习卡片，为后端 AI 卡片生成结果提供移动端查看入口

## 技术实现路径

Android AI 相关功能统一走以下分层：

```
Compose Screen
    ↓
ViewModel
    ↓
MainRepository
    ↓
ApiManager
    ↓
ApiService / Retrofit
    ↓
Backend AI API
```

这种结构让 AI 咨询、Agent 助手、客服对话和卡片列表都能复用统一的网络错误处理、Mock 切换和 Repository 门面。

## 遇到的问题和解决

1. **AI 功能类型多，协议不统一**：AI 咨询、Agent 工具调用、客服对话、卡片列表的响应结构差异较大。解决方式是为每类能力建立独立模型和 ViewModel，再在 Repository 层统一暴露用例。
2. **Agent 工具结果需要移动端友好展示**：后端工具结果包含结构化数据，直接展示不适合用户阅读。解决方式是在 Android 端增加工具名称映射、摘要展示、建议问题和 UI action 处理。
3. **后端接口联调存在不确定性**：AI 功能依赖后端和外部大模型能力，开发期间可能不可用。解决方式是保留 Mock 分支，确保页面交互、状态流转和测试可独立验证。
4. **多轮会话状态容易丢失**：Agent 和 AI 咨询都需要会话上下文。解决方式是在 ViewModel 中维护 `sessionId`，Agent 额外使用本地会话 store 管理多会话快照。

## 心得体会

AI 功能接入不是简单地把输入框接到聊天接口，而是要在移动端完成角色选择、会话管理、工具结果展示、异常兜底和后续操作引导。通过这部分工作，我对 AI 能力在客户端中的职责边界有了更清楚的认识：后端负责模型调用和工具编排，Android 端负责把结果组织成用户能理解、能继续操作的体验。
