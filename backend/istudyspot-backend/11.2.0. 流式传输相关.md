这个问题非常关键。你现在进入的是：

> **AI系统通信架构（AI Interaction Architecture）**

传统接口和 AI 接口，在“响应模型”上有明显区别。

你可以先建立一个整体地图。

------

# 一、传统API：请求 → 等待 → 返回

传统Web接口通常是：

```text
Request
↓

Server Processing
↓

Response
```

例如：

登录：

```text
POST /login
```

服务器：

- 查数据库
- 校验密码

然后：

一次性返回：

```json
{
  "success":true
}
```

这叫：

> Request / Response
> 同步响应模型

特点：

- 请求一次
- 返回一次
- 连接结束

前端：

```js
await fetch()
```

等结果。

------

这种模型适合：

- CRUD
- 数据查询
- 登录
- 订单

因为：

> 服务器处理时间短。

通常：

```text
50ms~2s
```

------

但AI不同。

------

# 二、AI为什么喜欢流式（Streaming）

AI生成：

尤其：

- LLM文本
- 图片生成
- 多步骤推理

特点：

不是：

```text
瞬间算完
```

而是：

```text
逐步生成
```

例如：

LLM：

实际上：

```text
Token
↓

Token
↓

Token
```

在生成。

ChatGPT看到的：

不是：

> 5秒后一次性冒出全文

而是：

> 字一个字冒出来。

因为：

模型本来就是：

> 自回归生成（Autoregressive）

天然流式。

------

所以：

AI系统常用：

> Streaming Response

而不是传统响应。

------

大概分：

三种模式。

------

# 三、模式1：同步一次性返回（Blocking）

最简单。

前端：

```text
POST /api/card/generate
```

后端：

1 调AI

2 等AI

3 全部完成

4 一次返回

：

```json
{
  "markdown":"",
  "imageURL":""
}
```

流程：

```text
Request
↓

AI生成
↓

完成
↓

Return
```

------

优点：

简单。

------

缺点：

用户：

```text
等待10~40秒
```

体验差。

容易：

- 超时
- 白屏
- 不知道在干什么

------

适合：

小Demo。

不适合AI产品。

------

------

# 四、模式2：流式传输（Streaming）

这是ChatGPT式。

核心：

> 连接不关闭，边生成边发送。

不是：

```text
一次返回
```

而是：

```text
持续推送
```

流程：

```text
Request
↓

Open Connection
↓

Token1
↓

Token2
↓

Token3
↓

Finish
```

前端：

实时显示。

------

常见协议：

两种。

------

## 方案A：SSE

Server-Sent Events

最常见。

很多AI系统都这样。

------

HTTP连接：

不关闭。

服务器：

持续：

```text
data:
```

发送。

例如：

服务器：

```text
data: {"text":"雨"}

data: {"text":"夜"}

data: {"text":"与"}

data: {"text":"灯火"}

data: [DONE]
```

前端：

：

```js
EventSource
```

或：

```js
fetch + stream reader
```

实时读。

效果：

```text
雨
雨夜
雨夜与
雨夜与灯火
```

逐步出现。

------

优点：

简单。

HTTP即可。

无需WebSocket。

------

适合：

AI文本。

------

ChatGPT早期：

大量使用。

------

------

## 方案B：WebSocket

双向长连接。

不是：

HTTP请求。

而是：

```text
Persistent Socket
```

长期连接。

双方：

都能主动发。

流程：

```text
Client ⇄ Server
```

实时。

------

例如：

AI陪伴、多人房间：

服务器：

```json
{
  "type":"token"
}
```

前端：

还能：

```json
{
  "type":"stop"
}
```

让AI停。

------

适合：

高互动场景。

例如：

- 实时聊天
- 语音
- 多人协作
- AI Agent

------

复杂度：

高。

------

你的卡片系统：

一般不需要。

------

------

# 五、模式3：异步任务（Job Queue）

这个在AI生图里非常常见。

因为：

图片：

通常：

```text
5~60s
```

甚至更久。

很多平台：

不保持连接。

而是：

> 提交任务 → 轮询结果

流程：

------

第一步：

提交任务：

```text
POST /generate
```

返回：

```json
{
  "taskID":"abc123"
}
```

这里只表示：

> 收到任务。

不是完成。

------

服务器：

后台：

```text
AI生图
```

运行。

------

前端：

继续：

轮询：

```text
GET /task/status
```

例如：

：

```json
{
  "status":"processing",
  "progress":40
}
```

最终：

：

```json
{
  "status":"done",
  "imageURL":""
}
```

------

流程：

```text
Submit
↓

Task Queue
↓

Poll
↓

Done
```

------

优点：

稳定。

适合：

- 生图
- 视频
- 长任务

------

Midjourney：

本质上类似。

------

# 六、AI系统常见架构

你可以形成这个认知：

| 类型      | 通信模式  |
| --------- | --------- |
| 登录/CRUD | 同步      |
| LLM聊天   | 流式SSE   |
| AI生图    | 异步任务  |
| 语音AI    | WebSocket |

------

所以：

你卡片系统：

其实有两个AI。

------

文本：

LLM。

天然：

> Streaming

图片：

生图。

天然：

> Async Job

------

这意味着：

卡片系统可能是：

混合架构。

------

# 七、结合你的卡片系统怎么选

我们分析：

------

文本：

AI Markdown。

可能：

2~8秒。

可以：

### 方案1

一次性返回。

简单。

或者：

### 方案2

SSE。

实时出现文字。

有：

“卡片正在写成”的体验。

------

图片：

即梦生图。

通常：

较慢。

推荐：

异步任务。

------

于是：

卡片生成可能：

------

方案A：

简单版：

同步。

：

```text
Generate
↓

等待
↓

一次返回
```

V1最容易。

------

方案B：

推荐版：

混合。

：

```text
创建任务
↓

SSE文本流
↓

图片异步
↓

最终完成
```

体验明显更高级。

------

你现在可以开始思考一个问题：

> **卡片生成在用户体验上，希望是“等待完成”，还是“生成过程可见”？**

这会直接决定：

你的 `/api/card/generate` 是同步接口，还是任务流接口。
